# デモ台本: Claude Code コンテキストマネジメント

## 前提

- **題材**: 小さなTodoアプリ（既存のプロジェクト）
- **所要時間**: 約15〜20分
- **準備**: Claude Code がインストール済みのターミナルを用意

---

## シーン1: コンテキストの確認（Part 1 対応）

### 目的
コンテキストウィンドウという概念を体感してもらう

### 手順

```
1. ターミナルで claude を起動
2. 簡単な質問をする
   → 「このプロジェクトの構成を教えて」
3. Claudeがファイルを読み込み、回答する
4. /context を実行
```

### 話すこと

> 「いま `/context` を打ちました。ここに表示されているのが、Claudeが今見えている世界の全体です。
> さっきの質問と回答、読み込んだファイル、それだけでもうこれだけ消費しています。
> さらに、CLAUDE.md やシステム指示など、我々が送っていないものも裏で読み込まれています。」

---

## シーン2: Plan Mode で予防する（Part 3 対応）

### 目的
「まず膨らませない」ことの重要性を示す

### 手順

```
1. /clear でリセット
2. Shift+Tab を2回押して Plan Mode に切り替え
   （画面に Plan Mode 表示を確認）
3. 「Todo に優先度機能を追加したい。まず現在のデータ構造を調べて」と入力
4. Claude が読み取り専用で調査・計画を提示
5. 計画に問題なければ Shift+Tab で Normal Mode に戻す
```

### 話すこと

> 「Plan Mode では Claude はファイルを読むだけで、書き込みはしません。
> いきなりコードを書かせると、間違った方向に進んだときの試行錯誤がすべてコンテキストに残ります。
> 先に Plan Mode で調べて計画を立て、合意してから実装する。これが予防の基本です。」

---

## シーン3: /compact で圧縮する（Part 4 対応）

### 目的
圧縮コマンドの使い方と焦点指定を見せる

### 手順

```
1. （Plan Mode の続きから）Normal Mode で実装を開始
   → 「では計画に沿って優先度機能を実装して」
2. 実装が進んだら /context で使用量を確認
3. /compact を実行
4. /context で圧縮後の使用量を確認
5. 差を見せる
```

### 話すこと

> 「`/compact` で会話を圧縮しました。ビフォーアフターを見てください。
> これだけ減りました。焦点を指定することもできます。
> 例えば `/compact 優先度機能のデータ構造の変更に焦点を当てて` と打つと、
> その部分を重点的に残して他を圧縮してくれます。」

---

## シーン4: /btw でサイド質問（Part 4 対応）

### 目的
メイン会話を汚さずに確認する方法を見せる

### 手順

```
1. Claude が実装作業中（ファイルを編集している途中）の状態で
2. /btw Priority enum の値は何がある？
3. Claude の作業が中断されず、オーバーレイで回答が表示される
4. オーバーレイを閉じると、Claude の作業がそのまま続いている
```

### 話すこと

> 「いま Claude が作業中ですが、`/btw` で割り込みました。
> ポイントは2つ。まず、回答はオーバーレイ表示で、メインの会話履歴には追加されません。
> そして、Claude のメイン作業は中断されず、そのまま続いています。
> ちょっとした確認をしたいだけなのに、メインの流れを壊したくないときに便利です。
> ただし、ツールは使えないので、既にコンテキストにある情報への質問に限られます。」

---

## シーン5: Rewind で巻き戻す（Part 4 対応）

### 目的
間違った方向に進んだときのリカバリを見せる

### 手順

```
1. わざと間違った指示を出す
   → 「やっぱり優先度は文字列じゃなくて数値にして」
2. Claude が修正する
3. 「やっぱり元の設計のほうがよかった」
4. Esc+Esc でrewindメニューを開く
5. チェックポイント一覧を見せる
6. 「Restore code and conversation」で復元
```

### 話すこと

> 「`Esc+Esc` で rewind メニューが開きます。ユーザーの入力ごと、
> ファイル編集の前にチェックポイントが自動で作られています。
> ここで Restore code and conversation を選ぶと、コードと会話の両方が巻き戻ります。」

---

## シーン6: Summarize from here（Part 4 対応）

### 目的
部分的な圧縮ができることを見せる

### 手順

```
1. （巻き戻した後、しばらく作業して会話が溜まった状態で）
2. Esc+Esc でrewindメニューを開く
3. 設計議論の直後のチェックポイントを選ぶ
4. 「Summarize from here」を選択
5. /context で確認 → 設計議論は残り、実装の試行錯誤だけ圧縮されている
```

### 話すこと

> 「`/compact` は会話全体を圧縮しますが、Summarize from here なら範囲を選べます。
> 設計の議論は残したまま、デバッグの試行錯誤だけ圧縮する、といった使い分けができます。」

---

## シーン7: /clear と Resume（Part 4 対応）

### 目的
完全リセットとセッション再開の流れを見せる

### 手順

```
1. 「優先度機能の実装が一段落した。次はフィルタ機能を追加したい」と宣言
2. /clear を実行
3. 「フィルタ機能を追加して」と新しいタスクを開始
4. 少し進めた後、「やっぱりさっきの優先度機能のセッションに戻りたい」
5. Ctrl+C で一旦抜ける
6. claude --resume でセッション一覧を表示
7. 前のセッションを選択して再開
```

### 話すこと

> 「`/clear` はコンテキストを完全にリセットします。でも CLAUDE.md は再読み込みされるので、
> プロジェクトのルールは消えません。
> 公式では『同じ問題で2回以上やり取りして解決しなければ /clear』を推奨しています。
> そして `claude --resume` で過去のセッションに戻れます。
> セッションをgitブランチのように扱う感覚ですね。」

---

## シーン8: Worktree で並列作業（Part 5 対応）

### 目的
セッション分離による並列開発を見せる

### 手順

```
1. ターミナルを2つ用意

ターミナル1:
   claude -w feature-filter

ターミナル2:
   claude -w bugfix-todo-delete

2. それぞれのターミナルで別々のタスクを開始
   ターミナル1: 「フィルタ機能を実装して」
   ターミナル2: 「Todoの削除が動かないバグを修正して」

3. 両方が独立して動いていることを見せる
4. 「今のディレクトリを教えて」と聞いて、ディレクトリが異なることを確認
```

### 話すこと

> 「`claude -w` で worktree を作ると、ディレクトリもブランチもセッションもすべて独立します。
> 機能開発の途中でバグ修正が入っても、コンテキストが混ざりません。
> ただし Auto Memory は共有されるので、一方で学んだことは他方にも反映されます。」

---

## シーン9: CLAUDE.md と Auto Memory（Part 6 対応）

### 目的
永続コンテキストの仕組みを見せる

### 手順

```
1. /clear でリセット
2. 「CLAUDE.md の内容を見せて」と Claude に依頼
   （なければ /init で生成）
3. CLAUDE.md に手動でルールを追記する例を見せる
   → 例: 「コミットメッセージは日本語で書く」
4. /clear → 新しいセッションでもルールが適用されることを確認

--- Auto Memory のデモ ---

5. 「このプロジェクトのパッケージマネージャは pnpm だと覚えておいて」と言う
6. /memory で Auto Memory の内容を確認 → 保存されていることを見せる
7. Ctrl+C で Claude を終了
8. claude を再起動
9. 「依存パッケージを追加するにはどうすればいい？」と質問
   → Claude が pnpm を前提に回答する（覚えている）
```

### 話すこと

> 「CLAUDE.md はあなたが書く永続的な指示です。セッションが変わっても毎回読み込まれます。
> `/init` で自動生成もできますが、200行以下を目指してください。長すぎると遵守率が下がります。
>
> 一方、Auto Memory は Claude が自動的に学んだことをメモする仕組みです。
> 『覚えておいて』と言えば明示的に保存もできます。
> ここでは『パッケージマネージャは pnpm だ』と覚えさせました。
> `/memory` で確認すると、ちゃんと記録されています。
>
> 一度 Claude を終了して再起動します。新しいセッションです。
> 『依存パッケージを追加するには？』と聞くと… pnpm を前提に回答してくれました。
> セッションをまたいで知識が引き継がれています。
>
> CLAUDE.md は『ルール』、Auto Memory は『学習ノート』と覚えてください。」

## クロージング

### 話すこと

> 「今日紹介したコマンドをまとめます。（早見表スライドを表示）
>
> 大事なのは、すべてを一度に覚える必要はないということです。
>
> まずは `/context` でコンテキストの使用量を意識すること。
> 次に Plan Mode で探索と実装を分けること。
> タスクが変わったら `/clear`、長くなったら `/compact`。
>
> この3つだけで、Claude の応答品質はかなり変わります。
> ぜひ明日から試してみてください。」
