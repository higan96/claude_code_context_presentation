# 今回のプレゼンテーション作資料成のプロンプト

## 最初のプロンプト

```
プレゼンテーションを作りたい。

内容は社内向けのClaude Codeの使い方、コンテキストマネジメント編。
この資料を読んで、コンテキストマネジメントの重要さ、コツがわかることを目指す。

主にはコンテキストとはなにか？なぜマネジメントが必要か？どうすればマネジメントできるか？具体的に利用するコマンドの使い方、rewind, resume, worktree, btwなど。

基本的に参照するのはClaudeCodeの公式ドキュメントのみにしたい。資料集めからはじめて、ソースとなる情報をマークダウンファイルに落とし込んで行きたい。計画を建てて。
```

## Claude Codeのアウトプット

プレゼンテーション作成計画

フェーズ1: 情報収集

公式ドキュメントから以下のトピックに関する情報を収集し、マークダウンファイルに保存する：

1. コンテキストウィンドウの基本概念 — コンテキストとは何か、トークンの仕組み
2. コンテキストマネジメントの必要性 — コンテキストが大きくなると何が起きるか、パフォーマンスへの影響
3. 具体的なコマンド・機能
  - /compact — コンテキストの圧縮
  - rewind — 会話の巻き戻し
  - resume — セッションの再開
  - Worktree — 並列作業のための分離環境
  - サブエージェント（btw含む） — コンテキスト分離による効率化
  - CLAUDE.md / メモリ — 永続的なコンテキスト管理

フェーズ2: 資料整理

収集した情報をソースファイル（sources/ディレクトリ）に整理

フェーズ3: プレゼンテーション作成

スライド構成を設計し、プレゼン資料を作成

## Claude CodeのアウトプットをCodexがレビュー

大筋はよいです。ただし、そのまま進めると後半で詰まりやすい点があります。先に直したほうがいいポイントを挙げます。

指摘

- 最重要: フェーズ1に「収集基準」がないです。今回は「Claude Code公式ドキュメントのみ」が条件なので、最初に 対象ドメイン、引用ルール、1主張=1出典 を決めないと、後でスライドの根拠が曖昧になります。
- 重要: トピックの粒度が少し混ざっています。コンテキストとは何か と コンテキストをどう減らすか と セッション運用 と 並列作業 が同列に並んでいるので、プレゼンでは理解の階段が作りづらいです。
- 重要: btw は用語として何を指すかを先に固定したほうがいいです。略称や社内呼称が混ざると、公式資料ベースの説明とズレる可能性があります。
- 重要: フェーズ2の「資料整理」が弱いです。単に sources/ に置くだけではなく、各mdに 要点, 原文要約, 使える図解案, スライド候補メッセージ を持たせると後工程がかなり楽になります。
- 中: 成果物の定義が不足しています。最終的に 収集メモ と スライド構成案 と 本編スライド を分けるかどうかを先に決めるべきです。
- 中: 検証観点がないです。社内向け資料では「そのコマンドを使うと何が改善されるか」を示したいので、各コマンドに 効く場面, 効かない場面, 誤用リスク を持たせると説得力が上がります。

改善した計画

1. 調査設計
   公式ドキュメント限定, 対象URL一覧, 引用フォーマット, 用語表 を先に作る。
2. 情報収集
   以下の章立てでソースを集める。
   コンテキストの基本
   なぜ管理が必要か
   管理手段: compact/rewind/resume
   作業分離: worktree/サブエージェント
   永続コンテキスト: CLAUDE.md/メモリ
3. ソース整理
   sources/*.md をトピック別に作り、各ファイルに 要旨, 根拠URL, スライドで使う主張候補, デモ候補 を書く。
4. ストーリー設計
   スライドを 概念 -> 問題 -> 解決策 -> コマンド例 -> 運用パターン の順に並べる。
5. プレゼン化
   まずは outline.md を作り、その後に本編スライドへ落とす。

おすすめのディレクトリ構成

sources/
  00_source_index.md
  01_context_basics.md
  02_why_management.md
  03_session_controls.md
  04_parallel_work.md
  05_persistent_context.md
slides/
  outline.md
  draft.md

この計画なら進めやすいです。特に最初に 00_source_index.md を作って、参照する公式ページ一覧を固定してから収集を始める形にするとブレません。必要なら次に、この計画をそのまま実行できる形で sources/ 用のテンプレート
まで作ります。

指摘

  (https://code.claude.com/docs/en/memory#choose-where-to-put-claude-md-files)
- 重要 sources/05_persistent_context.md:76 の Auto Memory のスコープ説明は、今回のテーマだと不十分です。原稿は「マシンローカル」で止まっていますが、公式は git リポジトリ由来の project ごとに 1 つ で、同じ repo
  の worktree は auto memory を共有する と書いています。今回 worktree を扱うなら、「worktree はセッション/作業ディレクトリを分けるが、auto memory までは分離しない」を明示しないと、コンテキスト分離の理解が崩れま

  Best practices (https://code.claude.com/docs/en/best-practices#manage-context-aggressively), Common workflows (https://code.claude.com/docs/en/common-workflows#use-plan-mode-for-safe-code-analysis)

## Claudeが修正、Codexで再レビュー

 指摘

- 重要 sources/03_session_controls.md:113 と slides/outline.md:142 の /btw 説明は、まだ少し言い切りが強いです。公式は「会話履歴に入らない」「追加コストは最小限」「親会話の全体は見える」と書いていて、コンテキスト

  モが残っていると後で再流用してズレが戻る可能性があります。

前回の主要な指摘はかなり解消されています。特に /btw とサブエージェントの対比、Auto Memory と worktree の関係、Plan Mode を使った「膨らませない」話の追加は、プレゼンの目的にかなり合っています。
