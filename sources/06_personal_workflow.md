# 個人的なワークフロー: compact / clear / rewind の使い分け

## 要旨

3つのコマンドを「いつ使うか」の判断基準を、実際の開発サイクルに沿って整理したもの。公式ドキュメントの推奨に加え、実践で得た運用パターン。

## ワークフロー

### /compact → コミットしたタイミング

- コミット = 1つの作業単位が完了した区切り
- そこまでの試行錯誤（ファイル読み込み、テスト実行、修正の繰り返し）はもう不要
- `/compact` で圧縮して、次の作業単位に集中できるコンテキストを確保する
- 焦点指定の例: `/compact focus on the committed changes and remaining tasks`

### /clear → 別の作業に切り替えたいとき

- 今の作業が完全に終わり、別のタスクに移る場面
- 前の作業のコンテキストは新しいタスクにとってノイズでしかない
- `/clear` でまっさらにして、新しいタスクに最大限のコンテキストを使う

### /rewind → プロンプトを間違えた・補足が必要だったとき

- 送信した直後に「あ、条件を書き忘れた」「指示が曖昧だった」と気づいた場面
- Claudeが間違った方向に進んでしまった場合も同様
- `Esc + Esc` で巻き戻して、正しいプロンプトでやり直す
- コンテキストに「失敗した試行」を残さないのがポイント

## スライドで使う主張候補

1. **commit → /compact、タスク完了 → /clear、ミス → /rewind** — 開発サイクルの自然な節目に対応
2. 公式のルールを暗記するより、自分の開発リズムに合わせて習慣化するほうが定着する

## 図解案

```
[コーディング] → commit → /compact → [次の作業単位]
                                          ↓
                                    全部終わった → /clear → [別タスク]

[プロンプト送信] → 「あ、間違えた」 → Esc+Esc → /rewind → [正しいプロンプトで再送]
```
